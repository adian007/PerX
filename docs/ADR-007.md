# ADR-007: Multi-Provider Packages, Currency/Localization, and Simulated Payments

**Status:** Accepted
**Date:** 2026-06
**Deciders:** Full team
**Context:** Gap analysis against TeamSystem Challenge 2026 brief, post-ADR-001 through ADR-006

---

## Context

The original six ADRs were written before the full challenge brief was reviewed in detail. Three requirements explicit in the brief are not currently modeled:

1. *"A single deal or a package that may span several providers"* + *"payment to one or many providers"* — our `plan_optimize` knapsack (ADR-004) optimizes budget fit, not multi-provider bundling, and there's no entity representing a "package" as a single unit spanning multiple provider selections.
2. *"Architect the product so it is not locked to one country... multiple languages, multiple currencies"* with **Albanian Lek** named explicitly — no currency or locale field exists anywhere in `001_initial.sql`.
3. *"The app routes a simulated payment to the relevant provider(s) and confirms the benefit"* — no `payments` table exists; `perk_selections.status` jumps straight from `approved` to implicitly "done" with no payment-confirmation step or evidence trail.

This ADR proposes the minimum schema and architecture changes to close these gaps **without restructuring anything already built**, given the time constraint of a 48-hour build.

---

## Decision

**Add three things, additively: a `currency`/`locale` layer, a `packages` entity, and a `payments` table.** No existing table is renamed, dropped, or has a column removed. All decisions below favor "smallest change that satisfies the brief" over "architecturally pure."

---

## Part 1 — Currency & Localization

### Decision
Add `currency_code` (ISO 4217, e.g. `ALL`, `EUR`, `USD`) to every money-bearing table, and `locale` to `users`. Store all prices as integer minor units (already your convention per `CLAUDE.md` — `*_cents` naming) but rename the suffix convention to `*_minor_units` going forward since Lek doesn't use cents-style subdivision the way EUR/USD do in practice (1 Lek has no commonly-used 1/100 subdivision in daily use, though qindarka technically exists). Simpler: **keep existing `_cents` columns as-is to avoid a rename under time pressure**, and treat the integer as "smallest unit per that currency's convention" — document this explicitly rather than renaming columns this late.

### Schema changes
```sql
ALTER TABLE perks ADD COLUMN currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL';
ALTER TABLE budget_allocations ADD COLUMN currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL';
ALTER TABLE perk_selections ADD COLUMN currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL';
ALTER TABLE employer_organizations ADD COLUMN default_currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL';
ALTER TABLE users ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'sq-AL';
```

### Consequences
- **No currency conversion logic is implemented.** A given employer org operates in one currency; a perk's `currency_code` must match the employer's `default_currency_code` at selection time, enforced at the service layer (not the DB) — a `CHECK` constraint across tables isn't practical in Postgres without a trigger, and a trigger is overkill for 48 hours.
- Frontend reads `currency_code` and formats display via `Intl.NumberFormat(locale, { currency: currency_code })` — zero backend i18n logic needed, JS's built-in API does the formatting work for free.
- `locale` defaults to `sq-AL` (Albanian/Albania) per the brief's "design the primary experience around Albania" instruction, while remaining swappable per-user — this is the cheapest possible way to satisfy "architected to scale across markets" without building a translation pipeline.
- **Explicitly out of scope for this hackathon:** live FX rates, multi-currency budgets per employer, translated perk *content* (descriptions stay in whatever language the provider entered them in). Flag this as a known limitation in the demo narrative rather than hiding it — judges weight "architected to scale" at 10%, not "fully localized today."

## Part 2 — Multi-Provider Packages

### Context on the distinction
`plan_optimize` (ADR-004) answers *"given my wishlist, what fits my budget?"* — a constraint-satisfaction problem over independently-priced items. The brief asks for something different: *"smart bundling of offers into packages"* — a curated *product*, where the bundling itself (not just the budget fit) is the value being delivered, often pre-assembled rather than employee-built.

### Decision
**Add a `packages` table as a thin wrapper over existing `perks`, not a new pricing engine.** A package is a named, curated set of perk references (potentially from different providers) that gets selected and approved as one unit.

### Schema
```sql
CREATE TABLE packages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    category VARCHAR(50),                       -- e.g. 'wellness_bundle', 'travel_bundle'
    is_curated BOOLEAN NOT NULL DEFAULT true,    -- true = admin/AI-assembled, false = employee-built
    created_by_user_id UUID REFERENCES users(id), -- null if system/AI-generated
    total_price_cents INTEGER NOT NULL,          -- sum of constituent perks at creation time
    currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL',
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

CREATE TABLE package_items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    package_id UUID NOT NULL REFERENCES packages(id) ON DELETE CASCADE,
    perk_id UUID NOT NULL REFERENCES perks(id),
    quantity INTEGER NOT NULL DEFAULT 1,
    UNIQUE(package_id, perk_id)
);

-- perk_selections gains an optional package reference
ALTER TABLE perk_selections ADD COLUMN package_id UUID REFERENCES packages(id);
```

### How this connects to existing architecture
- **Curated packages** (`is_curated = true`): seeded at demo-prep time, e.g. "Wellness Starter" = gym membership (Provider A) + healthy meal subscription (Provider B). This is your fastest path to satisfying "package spanning more than one provider" for the demo — you don't need the optimizer to discover this combination, you hand-curate 3-5 good ones.
- **Employee-built packages** (`is_curated = false`): when `plan_optimize` returns an approved multi-perk set, optionally offer "save this as a package" — this reframes your *existing* knapsack output as a "package" in the UI/marketing sense, satisfying the brief with no new algorithm, per the prioritization I suggested earlier.
- **Approval flow change:** when `perk_selections.package_id IS NOT NULL`, the employer approval screen shows it as one card ("Wellness Starter — €45/mo, 2 providers") rather than N separate line items. One employer approval action approves all `perk_selections` rows sharing that `package_id` — a service-layer batch operation, not a new approval state machine.
- **Payment fan-out:** one package approval triggers N payment records (Part 3), one per distinct provider in the package — this is where "payment to one or many providers" becomes literally true rather than just a budget-fit coincidence.

### Consequences
- Recommendation engine (ADR-002) can optionally surface curated packages alongside individual perks — same `recs:{employee_id}:scores` Redis structure, just include `package` as a recommendable type alongside `perk`. Minor service-layer addition, no schema change needed there.
- This satisfies the "Innovation & creativity" and "Two-sided marketplace flow" criteria more directly than the knapsack alone did.

## Part 3 — Simulated Payments

### Decision
**Add a `payments` table as an audit/evidence trail, triggered automatically on approval — no real payment rail, no external API.** This exists purely to make the core loop's final step *visible and provable* to judges, per the brief's explicit minimum-demo requirement.

### Schema
```sql
CREATE TABLE payments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    perk_selection_id UUID NOT NULL REFERENCES perk_selections(id),
    provider_id UUID NOT NULL REFERENCES provider_profiles(id),
    employer_id UUID NOT NULL REFERENCES employer_organizations(id),
    amount_cents INTEGER NOT NULL,
    currency_code VARCHAR(3) NOT NULL DEFAULT 'ALL',
    status VARCHAR(20) NOT NULL DEFAULT 'pending', -- pending | completed | failed
    simulated BOOLEAN NOT NULL DEFAULT true,        -- always true; column exists so a real
                                                      -- payment rail can be added later without
                                                      -- a schema change, just a default flip
    processed_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);
```

### Flow
1. Employer approves a `perk_selection` (existing flow from the approval-workflow diagram, unchanged)
2. **New step:** `BackgroundTasks.add_task(create_simulated_payment, selection_id)` — fires after the existing `status=approved` write, per the same "never block the response" pattern ADR-003 already established for the LLM
3. `create_simulated_payment` inserts a `payments` row with `status='pending'`, waits a short artificial delay (1-3s, purely for demo visual effect — "processing payment..." → "paid"), then updates to `status='completed'`
4. Employee's notification/selection-status screen shows the payment confirmation, satisfying *"confirms the benefit"* from the brief
5. For a package (Part 2), this fires once per distinct provider in the package, fanning out N payment rows from one approval

### Consequences
- **This is explicitly theater, and that's fine** — the brief says *"simulate payments (no real payment rail is required)."* The goal is a believable, demoable confirmation step, not payment infrastructure.
- `simulated` defaults to `true` and is never flipped in this project's scope — it exists so the schema doesn't need to change if a future, non-hackathon version integrates a real provider (Stripe Connect, a local Albanian payment processor, etc.). This is the one piece of "future-proofing" worth the few minutes it costs, since it costs nothing now and avoids a painful migration later.
- Employer analytics (`/employer/analytics`) can now show **real spend data** sourced from `payments` rather than inferring it from `perk_selections` status — slightly more accurate and gives you an extra "payments processed: €X this month" stat that's an easy, concrete demo beat.

## What this ADR deliberately does NOT do

- No real currency conversion or FX rates
- No multi-language *content* translation (only locale-aware number/date formatting)
- No real payment gateway integration of any kind
- No change to the knapsack algorithm itself — packages are a presentation/grouping layer on top of existing selection mechanics
- No new approval state machine — package approval is a batch operation over the existing states
